## 2) 数据规模怎么说（重点：真实、区间化、可解释）

你可以用“**当前落地规模 + 预留扩展规模**”说法，避免拍脑袋。

### 推荐口径（示例）
- 文档规模：`300~1000` 份（PDF/Word/Markdown 混合）
- 原始文本量：`几十万到几百万` 中文字符
- 分块后 chunk：`5k~30k`（取决于 chunk_size/overlap）
- 向量维度：与 embedding 模型一致（例如 768 / 1024 / 1536，按你实际模型说）
- 检索延迟：
  - 召回（向量+关键词）通常 `几十毫秒~几百毫秒`
  - 若开 rerank 和反思，端到端可能 `1~5s`（本地模型更慢）

### 如果怕被追问“你怎么估的？”
你就说：
- “按平均每份文档 N 页、每页 N 字、chunk_size=500 overlap=80 估算 chunk 数；上线后用 ingest 统计值校准。”

## 4) 面试高频追问（及答题思路）

### Q1: 为什么要“向量 + 关键词”两路召回？
- 语义检索擅长“同义表达”，关键词擅长“精确术语/编号/制度条款”。
- 两路互补，RRF 融合能提升召回覆盖率。

### Q2: 为什么还要 rerank？
- 召回阶段追求“召回全”，会带来噪音；
- rerank 做精排，把最相关片段顶上来，减少无关上下文污染生成。

### Q3: self-reflection 的收益和代价？
- 收益：低置信度答案可自动修正，降低“看起来很自信但错”的风险；
- 代价：多一次或多次 LLM 调用，时延和成本增加；
- 策略：仅在低分触发，控制重试次数。

### Q4: 你怎么控制幻觉？
- 严格提示词“只基于上下文回答”；
- 上下文不足时明确说“无法确定”；
- 返回 sources，做到可追溯；
- 反思机制兜底。

### Q5: 为什么用 FAISS，不直接上 Milvus/ES？
- 本地/中小规模 MVP，FAISS 部署轻、迭代快；
- 规模上来后可平滑迁移 Milvus（向量）+ ES（关键词），架构已解耦。

### Q6: Elasticsearch 在你项目里怎么接？
- 关键词路抽象成接口，当前用 BM25 本地实现；
- 上 ES 后只替换 keyword retriever，不影响后面的融合、rerank、生成链路。

### Q7: 线上最常见问题？
- 首次模型下载慢（embedding/rerank）；
- CPU 模式时延高；
- 文档解析质量参差（尤其扫描 PDF）；
- 解决：缓存模型、异步预热、OCR、离线评测和监控。

知识图谱 和 LangGraph是什么关系

好问题，一句话说清楚：

---

## 核心区别

| | **知识图谱** | **LangGraph** |
|--|-------------|---------------|
| **是什么** | 存数据的**数据库** | 管流程的**框架** |
| **存什么** | 实体和关系（张三-同事-李四） | 任务步骤和状态（理解→规划→执行） |
| **干什么用** | 查关系、做推理 | 管Agent怎么跑、步骤怎么流转 |

---

## 打个比方

- **知识图谱** = 地图（告诉你哪条路通哪）
- **LangGraph** = 导航APP（告诉你现在走到哪步了，下一步往哪走）

**它俩能配合：**
> LangGraph执行"查天气"任务时，可以去知识图谱里搜"北京-天气-怎么样"这个关系

---

## 你简历里怎么扯上关系

**你现在做的：**
> LangGraph管任务流程（节点A→节点B→节点C）

**说成知识图谱：**
> "基于图结构管理任务状态流转，支持实体关系推理与多跳决策"

**本质一样，换个说法就行。**

---

## 面试被问"它俩啥关系？"

**标准回答：**
> "知识图谱是**数据层**的图，存事实关系；LangGraph是**控制层**的图，管执行流程。两者都是**图结构思维**，可以结合——LangGraph做任务调度时，可以查询知识图谱做推理决策。"

---

## 一句话总结

> **知识图谱 = 存知识的地图**
> **LangGraph = 用地图导航的司机**
> 
> 一个管"知道什么"，一个管"怎么执行"，但都是**图的那套逻辑**。

你简历里写LangGraph经验，等于**间接证明你有图思维**，学知识图谱很快。


关于neo4j的面试考点

好的，Neo4j面试常问的点，我按**出现频率**和**难度**给你排：

---

## 一、必考点（90%会问）

### 1. Cypher查询语法
**最常考写法：**

```cypher
// 创建节点
CREATE (p:Person {name: '张三', age: 30})

// 创建关系
CREATE (p)-[:WORKS_AT]->(c:Company {name: '阿里'})

// 查询张三的同事（多跳查询）
MATCH (zhang:Person {name: '张三'})-[:WORKS_AT]->(c:Company)<-[:WORKS_AT]-(colleague:Person)
RETURN colleague.name
```

**面试问法：**
> "写个查询：找张三的同事的同事"

**答案：**
```cypher
MATCH (zhang:Person {name: '张三'})-[:WORKS_AT]->()<-[:WORKS_AT]-()-[:WORKS_AT]->()<-[:WORKS_AT]-(friend)
RETURN friend.name
```

---

### 2. 图数据库 vs 关系数据库

| | MySQL | Neo4j |
|--|-------|-------|
| 存什么 | 表（行+列） | 节点+关系 |
| 查关系 | JOIN（慢，越跳越慢） | 遍历（快，多跳也快） |
| 适合 | 固定结构数据 | 关系复杂、多跳查询 |

**面试金句：**
> "MySQL查3层关系要3个JOIN，性能爆炸；Neo4j遍历3层关系，速度几乎不变"

---

## 二、常考点（60%会问）

### 3. 什么时候用Neo4j？

**适合：**
- 社交网络（好友的好友）
- 推荐系统（买过X的人也买Y）
- 知识图谱（实体关系推理）
- 风控（资金流转路径）

**不适合：**
- 简单CRUD（用MySQL更简单）
- 大量属性、少关系（图的优势发挥不出来）

---

### 4. 性能优化

| 手段 | 干啥的 |
|-----|--------|
| 索引 | `CREATE INDEX ON :Person(name)` 加速查询 |
| 限制遍历深度 | `[:KNOWS*1..3]` 防止无限递归 |
| 方向查询 | 有关系方向时指定方向，减少扫描 |

## 四、你可能被问的"坑"

| 问题 | 陷阱 | 正确答案 |
|-----|------|---------|
| "Neo4j能存多少数据？" | 说"无限"就错了 | 单机亿级节点，分布式版更大，但太贵 |
| "和Redis图结构比呢？" | Redis是缓存，不是图数据库 | Neo4j是持久化图数据库，支持复杂查询 |
| "为什么不用NetworkX？" | NetworkX是内存计算，不能持久化 | 小数据用NetworkX，生产用Neo4j |

## 四、Cypher建图语句（面试可用）

```cypher
// 1. 创建设备节点
CREATE (gt:设备 {名称: "燃气轮机GT-001", 型号: "GT-A型", 位置: "一号厂房"})

// 2. 创建部件节点
CREATE (br:部件 {名称: "主轴轴承", 型号: "BR-501", 供应商: "SKF"})
CREATE (bl:部件 {名称: "涡轮叶片", 型号: "BL-202", 寿命: "8000小时"})

// 3. 创建故障相关节点
CREATE (yx:故障现象 {描述: "异响", 严重程度: "中", 检测方式: "人工听诊"})
CREATE (zm:故障原因 {描述: "轴承磨损", 根因: "润滑不良"})
CREATE (wx:维修措施 {操作: "更换轴承", 工时: 4, 工具: "液压拉马"})

// 4. 创建关系
CREATE (gt)-[:包含部件 {数量: 2}]->(br)
CREATE (gt)-[:包含部件 {数量: 24}]->(bl)
CREATE (gt)-[:出现现象 {发现时间: "2024-03-15"}]->(yx)
CREATE (yx)-[:可能原因 {概率: 0.7}]->(zm)
CREATE (br)-[:导致原因]->(zm)
CREATE (zm)-[:解决方法]->(wx)
CREATE (wx)-[:需要零件 {数量: 2}]->(br)

// 5. 查询：GT-001异响的可能原因和解决方法
MATCH (gt:设备 {名称: "燃气轮机GT-001"})-[:出现现象]->(现象)-[:可能原因]->(原因)-[:解决方法]->(方案)
WHERE 现象.描述 = "异响"
RETURN 现象.描述, 原因.描述, 方案.操作
```
